# P3.express для проекта AdventureMaker

## Команда (4 человека)

Роли распределены так, чтобы каждый отвечал за свою зону и не было дублирования:

- Даниил — постановка задач, код-ревью, работа с репозиториями (Coroutine Flow), настройка API, навигация, domain-слой, документация.
- Лера — дизайн интерфейса и его реализация, разработка ViewModel.
- Олеся — верстка экранов и часть ViewModel.
- Тарек — работа с репозиториями и ViewModel.

## 1. Цель проекта

Наша задача — сделать работающее туристическое приложение с картой, точками интереса и базовыми социальными функциями.

Как понять, что цель достигнута: пользователь может:

- открыть карту;
- найти место;
- добавить его в избранное;
- войти в аккаунт;
- посмотреть профиль.

При этом приложение должно работать стабильно, без падений.

Важно: фокус на MVP — минимально необходимом, но рабочем функционале.

## 2. Объём проекта

Что обязательно должно быть:

- карта и поиск;
- избранное;
- авторизация (Firebase);
- профиль;
- экстренные номера;
- переводчик.

Дополнительно (если останется время):

- посты;
- отзывы;
- конвертер валют;
- маршруты.

Если начинаем не успевать — уменьшаем функциональность, но не жертвуем стабильностью.

## 3. Архитектура

Используем:

- Clean Architecture;
- MVVM;
- Koin.

Слои:

- presentation;
- domain;
- data.

Критично:

- не смешивать слои;
- не переносить бизнес-логику в UI.

## 4. План работ

Работа разбита на недельные спринты.

### Спринт 1 — базовая настройка

- настройка проекта;
- подключение DI;
- навигация;
- базовые экраны.

Результат: приложение запускается.

### Спринт 2 — авторизация

- интеграция Firebase Auth;
- экран входа;
- получение пользователя.

Результат: можно войти в приложение.

### Спринт 3 — карта и места

- подключение 2GIS;
- загрузка и отображение мест;
- поиск.

Результат: карта функционирует.

### Спринт 4 — избранное

- добавление и удаление;
- список избранного;
- фильтрация.

Результат: пользователь может сохранять места.

### Спринт 5 — утилиты

- оффлайн-переводчик;
- экстренные номера.

Результат: базовый набор вспомогательных функций.

### Спринт 6 — доп. функционал (опционально)

- посты;
- лента.

## 5. Риски

Основные потенциальные проблемы:

- API (2GIS, Firebase) может быть нестабильным или сложным в интеграции -> решение: предусмотреть mock-данные.
- Нехватка ресурсов: команда небольшая -> решение: не распыляться на второстепенные задачи.
- Проблемы с архитектурой (часто возникает при быстром развитии проекта) -> решение: регулярное код-ревью.

## 6. Распределение задач

- UI-разработка — экраны, Compose, навигация.
- Data-слой — Retrofit, Firebase, преобразование моделей.
- Фичи — избранное, инструменты, посты.
- Общая архитектура и контроль — сложные части (например, карта), согласованность решений.

## 7. Процесс работы

- ежедневные короткие созвоны (10-15 минут);
- обсуждение прогресса и блокеров;
- демонстрация промежуточных результатов каждые несколько дней.

Правила:

- разбивать задачи на небольшие;
- коммитить регулярно;
- не выпадать из процесса.

## 8. Контроль прогресса

Оцениваем:

- работает ли функциональность;
- есть ли критические ошибки;
- укладываемся ли в сроки.

Если нет — пересматриваем объём задач.

## 9. Завершение проекта

Проект можно считать завершённым, если:

- реализованы все обязательные функции;
- приложение устанавливается и запускается;
- отсутствуют критические баги.

### Ключевая установка

Основной приоритет — довести продукт до рабочего состояния.

Лучше ограниченный, но стабильный функционал, чем широкий набор незавершённых возможностей.
