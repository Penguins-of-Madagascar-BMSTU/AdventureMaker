## Важная оговорка

Методология **12-Factor** изначально описывает **серверные** веб-приложения. Для **мобильного клиента** часть факторов применима напрямую, часть — **частично или по аналогии**, часть помечена как **N/A**, т.е. не применимо в классическом виде.

## Таблица разбора

| № | Фактор | Пояснение | Статус | Доказательства в проекте |
|---|--------|------------|--------|----------------------------|
| 1 | Кодовая база | Один репозиторий — много развёртываний | **Да** | Один Git-репозиторий `AdventureMaker`; Android-проект в `frontend/AdventureMaker/` (`settings.gradle.kts`: модули `:app`, `:domain`, `:data`). Сборки под разные среды — из одной истории коммитов |
| 2 | Зависимости | Явно объявлять и изолировать зависимости | **Да** | Каталог версий `frontend/AdventureMaker/gradle/libs.versions.toml`; зависимости объявлены в `app/build.gradle.kts`, `domain/build.gradle.kts`, `data/build.gradle.kts`. Сборка через gradle Wrapper (`frontend/AdventureMaker/gradlew`, `gradle/wrapper/`) |
| 3 | Конфигурация | Не хранить настройки в коде, выносить в окружение | **Частично** | Секреты и ключи для `data` читаются из свойств gradle (`property(...)` в `data/build.gradle.kts`), локально — из `gradle.properties`, в CI — через `-P…` в workflow. Конфиг Firebase для клиента: `app/google-services.json` |
| 4 | Сторонние службы | Подключать прозрачно, без внесения изменений в код | **Частично** | Внешние системы (Firebase, 2GIS и др.) подключаются через репозитории в `data/src/main/java/com/example/data/` и ключи из конфигурации сборки, а не захардкожены в UI. Смена окружения всё равно требует корректных свойств и файлов |
| 5 | Сборка, релиз, запуск | Строго разделять эти три этапа |  |  |
| 6 | Процессы | Приложение не должно хранить внутреннее состояние | **N/A** | Для Android-клиента процесс хранит UI-состояние, кэш, `DataStore` и т.п. |
| 7 | Привязка портов | Экспортировать сервисы через привязку к порту | **N/A** | Мобильное приложение не поднимает HTTP-сервис на порту |
| 8 | Параллелизм | Масштабироваться через запуск дополнительных процессов | **N/A** | Не применимо |
| 9 | Утилизируемость | Быстрый запуск и корректное завершение процессов | **Частично** | Завершение процесса определяет ОС и действия пользователя, нет SIGTERM — есть колбэки жизненного цикла `onDestroy()` и др., но они не гарантированы при убийстве процесса системой |
| 10 | Паритет сред | Среда разработки должна быть максимально похожа на прод | | |
| 11 | Журналирование | Логи — поток событий (stdout), централизованный сбор | **Частично** | Используется `android.util.Log`, например, в `data`-слое. Это не классический поток только в stdout контейнера; нет единой системы логов в проекте |
| 12 | Задачи администрирования | Выполнять как разовые процессы | **Частично** | Разовые действия возможны через gradle `./gradlew …`. нет конкретных admin-скриптов в репозитории; не определено, какие административные задачи могут понадобиться |
